## Dependency notes

- **007 requires 006**: it reuses the trust-scope helper 006 exposes, and applies
  the same `<trust>/<repo>` scoping idea to the executable tool stores.
- **013 requires 012**: token-refresh classification must key on the typed
  `GitHubApiError.status`, not error-message substrings.
- **017 requires 016**: once `serde_yaml` is replaced, the `deny.toml` no longer
  needs a `serde_yaml` RUSTSEC ignore.
- **020 should follow 005**: the perf rewrite of the masking path must preserve
  005's multi-line + add-mask semantics.
- **029 requires 006 + 007**: GC must honor the trust/repo scoping those add.
- **018 unblocks 019 (loop-level) and 032**: the fake-server harness is the seam
  for protocol/cancellation regression tests. 011's panic-completion also gains
  an end-to-end test there.
- **031 relates to 009**: if the decision is to fail the step, route it through
  009's failed-step handling.
- **Soft coupling — cache path changes (006):** 002's staging dir inherits 006's
  scoping via `with_file_name`; if both are in flight, land 002 first to avoid a
  conflict in `save_cache_result`.

## Cross-cutting notes

- **Composability:** 001, 009, 010, 011, 013 all touch step/job outcome flow.
  Their tests overlap; run the full suite after each. 010's timeout and 011's
  panic-completion both rely on the job still *completing* to GitHub — keep that
  invariant.
- **Masking (005 + 020):** security-critical. Never weaken an assertion; 020 must
  prove output-equivalence to 005's semantics.
- **Roadmap alignment:** 012 introduces the typed error the planned native-HTTP
  client (master-plan P3.1) will need behind a transport trait (a deferred
  follow-up, not planned here). 020 is a stepping stone toward the P3.2
  `bytes::Bytes` log rebuild. 029 serves the stability-first mandate
  (GC-before-park = self-heal instead of park-and-wait).

## Findings considered and rejected (so they aren't re-audited)

- **Edition 2021 → 2024 bump**: mechanically cheap (no `impl Trait` return sites;
  2 test-only `set_var`) but unblocks nothing and is not EOL/security-driven.
  Not worth prioritizing; do opportunistically in a quiet window.
- **`rsa` Marvin advisory (RUSTSEC-2023-0071)**: usage is OAuth-JWT **signing
  only**, local, ephemeral key (`protocol.rs:479,858`); the decryption-oracle
  threat is not on this path and no fixed `rsa` release exists. Accept-and-
  document via `deny.toml` (plan 017), not migrate.
- **`.editorconfig` / `rustfmt.toml` / `clippy.toml` absence**: rustfmt/clippy
  defaults are enforced in CI; only a minimal `.editorconfig` would add value
  (low priority). Folded into general DX, not a standalone plan.
- **Doc-tests not run in CI**: currently zero doc-tests exist, so nothing is
  skipped today — a latent trap, not a present defect. Add a `cargo test --doc`
  step if/when doc examples are introduced.
- **Duplicate transitive crates other than sha2** (`getrandom` ×3, `rand`/`rand_core`
  ×2, `hashbrown` ×2): unavoidable transitive spread; not worth pinning. Only the
  direct `sha2 0.11` vs 0.10 split is addressed (plan 017).
- **Streaming artifact upload / in-memory zip OOM (P3.3)** and **org-level JIT +
  dynamic slot autoscaling (P3.4)**: real and valuable, but already the team's own
  roadmap items (master-plan Phase 3), not new findings. The seams exist
  (protocol scope for org JIT at `protocol.rs:105`; the in-memory zip at
  `protocol.rs:3230`). Left to the roadmap; not duplicated as plans here.
- **curl subprocess transport, per-line `String` allocation, fixed slot fleet,
  interim personal PAT**: documented, decided tradeoffs (master-plan). Excluded
  by design.
