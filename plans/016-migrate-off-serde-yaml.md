## Why this matters

`serde_yaml` 0.9 is **unmaintained and archived** (RUSTSEC-2024-0320); its
backend `unsafe-libyaml` is likewise archived. Velnor uses it to parse
`action.yml` / workflow YAML pulled from **checked-out third-party actions** —
an untrusted-input path — so a latent parser bug would never get an upstream
fix. Migrating to a maintained, API-compatible fork removes a permanently-
unpatched dependency from a security-relevant path. This is mechanical but wide:
56 call sites across 4 files.

## Current state

- Workspace dep declaration: `Cargo.toml:29` — `serde_yaml = "0.9.34"`.
- `Cargo.lock` — `serde_yaml 0.9.34+deprecated` + `unsafe-libyaml 0.2.11`.
- Call sites (`rg -c "serde_yaml::"`): `crates/velnor-runner/src/executor.rs` (13),
  `crates/velnor-runner/src/action.rs` (6), `crates/velnor-runner/src/runner.rs`
  (5), `crates/velnor-tools/src/main.rs` (32). Mostly `serde_yaml::from_str` /
  `serde_yaml::to_string` / `serde_yaml::Value`.
- Migration target: **`serde-yaml-ng`** — a maintained fork with the same
  `serde_yaml::` API surface (same `from_str`/`to_string`/`Value`), chosen as the
  closest drop-in. (Alternative: `serde_yml`; if `serde-yaml-ng` turns out not to
  be drop-in, evaluate `serde_yml` before hand-porting.)
- Safety net: the workspace has ~420 tests; YAML round-trips are exercised by
  action/workflow parsing tests. Use them to catch any serialization-format
  difference (tag/quoting/key ordering).

## Steps

### Step 1: Confirm the fork is drop-in

Check `serde-yaml-ng`'s docs/API (it re-exports the `serde_yaml` surface).
Confirm it exposes `from_str`, `to_string`, `Value`, `Mapping`, and any other
`serde_yaml::` item used in the 4 files (grep the exact items:
`rg "serde_yaml::\w+" -o crates/ | sort -u`). If any used item is missing from
the fork, STOP and report the item — do not hand-roll a shim.

**Verify**: the list of used `serde_yaml::` items all exist in the fork.

### Step 2: Swap the dependency

In `Cargo.toml:29`, replace `serde_yaml = "0.9.34"` with the fork. The
lowest-churn approach keeps the code's `serde_yaml::` paths working by renaming
the crate in the manifest:

```toml
serde_yaml = { package = "serde-yaml-ng", version = "<latest 0.10.x>" }
```

This way the 56 `serde_yaml::` call sites need **no** edit (the crate is renamed
via `package =`). If the team prefers an explicit rename, that is a larger diff;
prefer the `package =` alias to minimize risk. Regenerate the lockfile with a
normal `cargo build --workspace` (do not hand-edit `Cargo.lock`).

**Verify**: `cargo build --workspace --locked` → exit 0 (after the lock updates);
`grep -n 'serde_yaml\|unsafe-libyaml' Cargo.lock` shows the new fork and **no**
`0.9.34+deprecated` / `unsafe-libyaml`.

### Step 3: Run the full suite and check round-trips

**Verify**: `cargo fmt --all --check` → exit 0; `cargo clippy --workspace
--all-targets --locked -- -D warnings` → exit 0; `cargo nextest run --workspace
--locked` → all pass. If any YAML round-trip / action-parsing test fails due to a
formatting difference (quoting, tag style, key order), that is the migration's
real risk — read the diff, and if the fork's output legitimately differs in a
way that matters to GitHub parity, STOP and report (do not blindly update the
expected value).
