# Primary-source research: one CI contract for Velnor projects

Research date: 2026-07-18. This note is an input to `VELNOR_PROJECTS_SETUP.md`, not an implementation. Sources are first-party documentation or repository source.

## Recommended common contract

### 1. Treat runner mode as workflow data, not duplicated workflow logic

Every executable workflow should expose one `workflow_dispatch` choice named `lanes` with exactly `velnor`, `github`, and `both`, defaulting to `velnor`. Automatic events (`push`, `pull_request`, `schedule`, and applicable `workflow_run`) must resolve to `velnor`. A small metadata job should emit a JSON `runner-configs` matrix; every workload job consumes the same matrix and uses `runs-on: ${{ fromJSON(matrix.runner) }}`. GitHub documents that a matrix can be constructed from JSON job output and that `runs-on` accepts a string, an array, or group/label objects ([matrix jobs](https://docs.github.com/en/actions/how-tos/write-workflows/choose-what-workflows-do/run-job-variations), [runner selection](https://docs.github.com/en/actions/how-tos/write-workflows/choose-where-workflows-run/choose-the-runner-for-a-job)). Manual choice inputs resolve to strings and are available through `inputs` ([workflow dispatch](https://docs.github.com/en/actions/how-tos/write-workflows/choose-when-workflows-run/trigger-a-workflow)).

Use these two canonical matrix records (plus non-routing metadata such as cache-writer policy):

```json
{"lane":"Velnor","runner":"[\"self-hosted\",\"velnor-target-mvp\"]","runner_os":"Linux","runner_arch":"X64"}
{"lane":"GitHub","runner":"\"ubuntu-26.04\"","runner_os":"Linux","runner_arch":"X64"}
```

This matches the current `java-monorepo` policy: Velnor is the default; GitHub is an explicit, exactly pinned `ubuntu-26.04` comparison; `both` executes the same workload on both lanes; a mutating workflow must have only one writer ([repository policy](https://github.com/ChainArgos/java-monorepo/blob/main/.github/AGENTS.md)). `ubuntu-latest` must not be used: GitHub says `-latest` tracks the latest GA image and announces migrations, so it is an intentionally moving alias ([runner-images label scheme](https://github.com/actions/runner-images#available-images)). Note that GitHub presently lists `ubuntu-26.04` as public preview, while `ubuntu-latest` maps to 24.04; adopting the requested rule therefore requires Velnor to match the explicit 26.04 contract and the estate to accept preview-image churn ([GitHub runner table](https://docs.github.com/en/actions/how-tos/write-workflows/choose-where-workflows-run/choose-the-runner-for-a-job)).

Do not rely on labels alone to prove Ubuntu. GitHub explicitly says self-hosted default labels such as `linux` and `x64` are accepted without validating the actual machine ([self-hosted labels](https://docs.github.com/en/actions/how-tos/manage-runners/self-hosted-runners/apply-labels)). Velnor should expose immutable image/OS metadata and each common bootstrap should fail fast unless `runner.os == Linux`, `runner.arch == X64`, and the Velnor job image identifies the required Ubuntu release. Container and service-container jobs require Linux plus Docker ([self-hosted requirements](https://docs.github.com/en/actions/reference/runners/self-hosted-runners)).

Repository or organization variables are useful for non-secret configuration, but **not** as a hidden per-repository default that can drift. GitHub variables are deliberately mutable at organization/repository/environment scope and unset variables evaluate to an empty string ([variables](https://docs.github.com/en/actions/concepts/workflows-and-actions/variables), [contexts](https://docs.github.com/en/actions/reference/workflows-and-actions/contexts)). Encode the Velnor default in versioned workflow policy; reserve a variable for an explicit emergency/operator override only if its audit and expiry semantics are defined.

### 2. Reuse semantic jobs, not a giant generic workflow

Standardize a thin, versioned reusable-workflow layer for recurring job families (Rust quality/test, TypeScript quality/test/build, Docker build/publish, release), with typed inputs for project-specific commands and artifacts. Keep trigger definitions and repository-specific dependency graphs in each repository. GitHub requires reusable workflows under `.github/workflows` with `workflow_call`, supports typed inputs/secrets and matrix callers, and permits nesting ([reusable workflows](https://docs.github.com/en/actions/how-tos/reuse-automations/reuse-workflows)). Pin cross-repository reusable workflows to a full commit SHA just like actions.

The lane matrix must be passed to every workload job, not merely a top-level smoke job. Aggregate/required-check jobs should use `if: always()` and validate every selected lane. For deployment, releases, Renovate, cache publication, and any other external mutation, `both` must mean one designated writer and one build/verify-only lane. This avoids duplicate releases while preserving parity evidence.

### 3. Mise is the one provisioning front door

Commit `mise.toml` and `mise.lock`; install all user-space CI tools through a full-SHA pin of the current `jdx/mise-action@v4`, then invoke named `mise run ci:*` tasks from workflows. Mise’s official CI guide says its action installs mise and project tools and supports GitHub cache integration; the action automatically uses `mise install --locked` when a lock exists ([mise CI](https://mise.jdx.dev/continuous-integration.html), [mise-action lock and cache behavior](https://github.com/jdx/mise-action#lock-files)). Set an explicit mise version rather than `latest`, use lockfile verification, and make cache paths/keys lane-independent where contents are portable. On Velnor, the runner’s host-persistent mise store should make the action’s remote cache path a no-op; on GitHub, the same declared action should use Actions cache. Use `mise x -- ...` when a context does not load mise shims.

Avoid parallel `mise install` calls against one shared prefix. One bootstrap step owns installation; later jobs either restore a verified immutable tool bundle or invoke mise once in that isolated job. PR #810’s useful direction is a prepared bootstrap plus Rust-owned orchestration, but its stated policy that GitHub is automatic/default conflicts with this project’s Velnor-default requirement and must not be copied ([PR #810](https://github.com/jackin-project/jackin/pull/810)).

### 4. Rust has one checked-in source of truth

Each Rust repository should commit `rust-toolchain.toml` with an exact current stable channel, `profile = "minimal"`, and only required `components` (`rustfmt`, `clippy`) and cross targets. Rustup documents that the file is source-controlled, selects the toolchain automatically, and can declare profile/components/targets; the minimal profile is recommended for CI ([toolchain override file](https://rust-lang.github.io/rustup/overrides.html#the-toolchain-file), [profiles](https://rust-lang.github.io/rustup/concepts/profiles.html)). Keep `package.rust-version` aligned with that file where applicable. Mise may install `rustup`, but must not become a second independent Rust-version source.

Run `cargo fetch --locked` before compilation; use `--locked` for all CI Cargo commands. Cache downloaded inputs (`~/.cargo/registry` and `~/.cargo/git/db`) separately from compiled outputs. Keys must include the lockfile(s), Rust toolchain, target triple, relevant profile/RUSTFLAGS, and a versioned cache contract. A fallback registry cache must be verified with `cargo fetch --locked --offline` before trust. Do not cache credentials.

Use `sccache` as `RUSTC_WRAPPER` for every compiling job and publish end-of-job stats. The upstream project describes sccache as a compiler wrapper that avoids recompilation and supports remote backends ([mozilla/sccache](https://github.com/mozilla/sccache)); its maintained Actions integration currently documents `mozilla-actions/sccache-action@v0.0.10`, `SCCACHE_GHA_ENABLED=true`, and notes that sccache versions before 0.10 are likely to fail ([sccache action](https://github.com/mozilla-actions/sccache-action#usage)). Velnor should provide a trust-scoped persistent/remote sccache backend; GitHub jobs should use the corresponding remote backend, so both lanes exercise the same compiler-cache contract. Persistent `CARGO_TARGET_DIR` is an optional, more fragile optimization: scope by trust/repository/workflow/job/toolchain/target/profile, never share it across incompatible jobs, and require rerun audits to detect unexpected compilation. If an estate repo elects to use `Swatinem/rust-cache`, run toolchain setup first, configure every workspace, and disable its `~/.cargo/bin` caching/cleanup on warm Velnor hosts; its own documentation identifies this long-lived-self-hosted-runner hazard ([rust-cache details and known issues](https://github.com/Swatinem/rust-cache#known-issues)).

### 5. Node/TypeScript installs are lockfile-exact and cache the package store

Declare Node and the package manager (Bun/pnpm/npm/Yarn as actually used by the repository) in mise; commit the package-manager lockfile; use its immutable install mode (`bun install --frozen-lockfile`, `pnpm install --frozen-lockfile`, `yarn install --immutable`, or `npm ci`). Cache the package manager’s download/store directory, not a mutable cross-job `node_modules`, unless a repository proves a safe immutable result cache. GitHub’s cache reference says setup actions provide package-manager caching for npm/Yarn/pnpm, cache keys should incorporate OS and lockfile hash, cache contents are not signed, and secrets must never be cached ([dependency caching](https://docs.github.com/en/actions/reference/workflows-and-actions/dependency-caching)); `setup-node` likewise documents that it caches global package data rather than `node_modules` and recommends `cache-dependency-path` for monorepos ([setup-node cache semantics](https://github.com/actions/setup-node#caching-global-packages-data)). Because mise is the required provisioning front door, reproduce those cache semantics in the shared cache component rather than adding divergent language setup actions.

PR and low-trust jobs should restore trusted default-branch caches but not publish into privileged scopes. GitHub documents that fork PRs can read base-branch caches and that poisoned caches can lead to code execution; low-trust default-branch-scope triggers are read-only ([cache security](https://docs.github.com/en/actions/reference/workflows-and-actions/dependency-caching#cache-access-for-low-trust-workflow-triggers)). Velnor must enforce at least the same trust boundary for its persistent stores.

### 6. Docker uses BuildKit/Buildx with one cache identity per image

Use the same Dockerfile/build context, tags, build args, secrets, platforms, and provenance settings in both lanes. The current official action line is `docker/setup-buildx-action@v4` plus `docker/build-push-action@v7`; pin their verified commits. Set up current Buildx/BuildKit explicitly on self-hosted Velnor; Docker warns that GitHub-hosted images carry current Buildx/BuildKit but self-hosted runners may need updating ([Docker CI cache guidance](https://docs.docker.com/build/ci/github-actions/cache/)). Actions Cache API v2 is mandatory (v1 was shut down on 2025-04-15), requiring at least Buildx 0.21 and BuildKit 0.20 on self-hosted runners. Every image gets a stable unique cache scope; otherwise the default `buildkit` scope causes multiple images to overwrite one another. Use both `cache-from` and `cache-to`, with `mode=max` for the trusted main writer and restore-only or appropriately isolated `mode=min` for PRs/comparison readers. Docker documents cache scopes, `min`/`max`, GitHub cache v2, and the `docker/build-push-action` automatic runtime credentials ([GHA cache backend](https://docs.docker.com/build/cache/backends/gha/), [cache backends](https://docs.docker.com/build/cache/backends/)). For multi-provider release durability, consider a registry cache ref; `inline` supports only `mode=min`. Note that BuildKit cache-mount contents are not exported by `type=gha` without Docker's documented cache-mount workaround, while Velnor's persistent local builder can retain them ([cache mounts](https://docs.docker.com/build/ci/github-actions/cache/#cache-mounts)).

Velnor’s persistent local builder should remain the hot path, but it must interpret the same declarative cache contract; its native Docker adapters may omit redundant `type=gha` transfer only when this is visible in logs and parity tests. Build secrets must use BuildKit secret mounts, never `ARG`/`COPY`, because Docker warns those can leak into cache/images ([cache backends warning](https://docs.docker.com/build/cache/backends/)). Emit plain progress and cache-hit evidence for repeat-run auditing.

### 7. Security and reproducibility are part of the template

Pin every third-party action and reusable workflow to a verified full-length commit SHA, retaining the release tag in a comment for maintainability. GitHub says a full SHA is the only immutable action reference and supports organization/repository policies requiring SHA pinning ([secure use](https://docs.github.com/en/actions/reference/security/secure-use#using-third-party-actions), [permissions API](https://docs.github.com/en/rest/actions/permissions)). Use Dependabot/Renovate to update pins through review.

Start workflows with `permissions: contents: read` (or `{}` where checkout is unnecessary), then grant the least required permissions per job. GitHub recommends least privilege for `GITHUB_TOKEN`; if stronger capabilities are needed, prefer a GitHub App’s fine-grained, short-lived installation token over a user PAT ([GITHUB_TOKEN](https://docs.github.com/en/actions/tutorials/authenticate-with-github_token), [secrets](https://docs.github.com/en/actions/concepts/security/secrets)). Never expose secrets to untrusted fork code or shared caches. Add explicit `timeout-minutes`, ref-scoped concurrency/cancellation for validation, non-cancelling serialization for publication, and artifact attestations/provenance where releases require them.

## What PR #810 contributes—and what it does not

The PR demonstrates several transferable ideas: one readable complete job per semantic unit; a single metadata job producing a runner JSON matrix; exact-result proofs; separate registry and target-cache contracts; offline validation of restored Cargo inputs; named, diagnostic steps; bounded timeouts; full-SHA action pins; Rust-owned CI policy/transport/auditing rather than large embedded shell programs; and automated checks that warm runs contain no downloads or dependency compilation ([PR description and changed workflows](https://github.com/jackin-project/jackin/pull/810)). Those should inform the common estate template.

Do **not** copy its current lane policy verbatim: the PR defaults automatic CI to GitHub and makes Velnor manual, whereas this estate’s required contract is the inverse. Do not universalize its project-specific exact-result cache or one-crate/one-job topology either; apply the semantic-unit principle to each repository’s actual build graph. Its cache-save rule is also internally broader in implementation than the prose policy (the composite action saves for same-repository PR/dispatch); the estate standard should make the writer/trust policy executable and test it.

## Velnor capabilities required by the standardized workflows

1. Exact GitHub expression/matrix support for `workflow_dispatch` choice inputs, job outputs, `fromJSON`, arrays in dynamic `runs-on`, `strategy.matrix`, `needs`, `if: always()`, and reusable `workflow_call` inputs/secrets/outputs.
2. A Velnor Ubuntu 26.04 x64 job image contract matching the pinned GitHub comparison environment closely enough for parity, plus fail-fast reported OS/image identity. Labels alone are not proof.
3. Native latest-version adapters for the standardized pinned actions: checkout, cache restore/save, mise, Docker setup-buildx/login/metadata/build-push (or bake), artifacts, and any attestation actions actually adopted. Unsupported fixture/estate behavior must be implemented in Velnor, not removed from workflows.
4. Actions Results Service v2 cache and artifact semantics, including branch/trust visibility, restore keys, exact-hit outputs, immutable saves, concurrency, cancellation, and safe behavior when `both` runs simultaneously.
5. Host-persistent, trust-scoped mise, Cargo registry/git, sccache, optional Cargo target, package-manager, and BuildKit stores mapped to truthful container paths (`HOME=/github/home`). Cache isolation must include repository and trust scope; persistent target outputs require toolchain/target/profile/job boundaries.
6. Current Docker/Buildx/BuildKit, service containers, job containers, multi-platform builds, registry auth, BuildKit secrets, cache mounts, `type=gha` v2 semantics, provenance/SBOM, and readable plain progress.
7. Correct secret and permission boundaries in `both` mode: untrusted jobs cannot receive repository secrets, access the trusted Docker socket, poison privileged caches, or publish. A single-writer mechanism must prevent duplicate release/deploy/renovation effects.
8. Parity observability: lane identity on every job, cache hit/miss and sccache stats, toolchain/image versions, downloadable raw logs, and an automated warm-rerun audit proving no dependency/tool downloads or avoidable compilation.

## Acceptance gates for the eventual rollout

- Static audit: every executable workflow supports all three modes; automatic resolution is Velnor; GitHub resolves only to exactly `ubuntu-26.04` (or the policy’s explicit ARM exception); no `ubuntu-latest`; all action references are full SHAs.
- Behavioral audit: dispatch `velnor`, `github`, and `both`; the selected workload set and commands are identical modulo declared lane/cache-writer metadata; required aggregate checks cover all selected lanes.
- Environment audit: both lanes report Ubuntu 26.04 x64, pinned mise, identical Rust/Node/package-manager versions, lockfile-exact installs, and current Docker/BuildKit versions.
- Rerun audit: a just-completed unchanged pipeline downloads and recompiles nothing avoidable; logs prove Cargo/package/Docker/sccache hits rather than merely a green result.
- Trust audit: fork/untrusted runs receive no secrets or trusted socket, cannot write privileged persistent caches, and cannot mutate external state; `both` has exactly one writer.
- Velnor contract audit: any divergence is fixed in Velnor or the shared standard, never by weakening a repository workload or fixture.
